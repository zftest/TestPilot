## 一、权限模型：RBAC（基于角色的访问控制）

```
          ┌──────────────┐
          │   用户 User   │
          │  狼图腾OPC    │
          └──────┬───────┘
                 │ 拥有
          ┌──────▼───────┐
          │  角色 Role    │  一个用户可以拥有多个角色
          │  超级管理员    │  角色决定权限集合
          └──────┬───────┘
                 │ 关联
          ┌──────▼───────┐
          │  权限 Permission│
          │  project:create │
          │  case:read      │
          │  execution:execute│
          └────────────────┘

权限 = 资源(resource) + 操作(action)
例如: project:create = 对project资源的create操作
```

## 二、认证流程

### 2.1 登录流程

```
用户输入用户名+密码
    │
    ▼
POST /api/auth/login
{username, password}
    │
    ▼
后端验证 bcrypt(password_hash)
    │
    ├── 失败 → 登录失败计数+1 → 连续5次锁定30分钟
    │
    └── 成功 → 生成 JWT Token（含 user_id, roles, exp）
                ├── Access Token: 有效期 2小时
                └── Refresh Token: 有效期 7天
                        │
                        ▼
                返回前端，存储到 localStorage
                        │
                        ▼
                后续请求在 Header 携带：
                Authorization: Bearer <access_token>
```

### 2.2 Token刷新流程

```
Access Token 过期 → 前端收到 401
    │
    ▼
POST /api/auth/refresh
{refresh_token}
    │
    ▼
后端验证 Refresh Token
    ├── 有效 → 下发新的 Access Token + Refresh Token
    └── 过期 → 跳转登录页
```

### 2.3 权限校验中间件

```python
# backend/middleware/auth.py
from fastapi import Depends, HTTPException
from jose import JWTError, jwt

async def get_current_user(token: str = Depends(oauth2_scheme)):
    try:
        payload = jwt.decode(token, SECRET_KEY, algorithms=[ALGORITHM])
        user_id = payload.get("sub")
        if user_id is None:
            raise HTTPException(401, "无效Token")
        # 查询用户是否存在/是否被禁用
        user = await db.get_user(user_id)
        if user.status == "disabled":
            raise HTTPException(403, "账号已被禁用")
        return user
    except JWTError:
        raise HTTPException(401, "Token已过期")

def require_permission(permission_code: str):
    """权限装饰器"""
    async def dependency(user = Depends(get_current_user)):
        has_perm = await db.check_permission(user.id, permission_code)
        if not has_perm:
            raise HTTPException(403, f"缺少权限: {permission_code}")
        return user
    return dependency

# 使用示例
@router.post("/api/projects")
async def create_project(
    user = Depends(require_permission("project:create"))
):
    ...
```

## 三、角色定义

### 3.1 系统内置角色

| 角色编码 | 显示名 | 权限范围 | 典型人员 |
|----------|--------|---------|----------|
| super_admin | 超级管理员 | 全部权限，含系统配置 | 平台Owner |
| admin | 管理员 | 团队管理、项目管理、所有测试功能 | 团队Leader |
| pm | 项目经理 | 项目管理、报告查看、需求管理 | PM |
| tester | 测试工程师 | 用例设计、执行、报告查看 | QA |
| viewer | 观察者 | 只读访问 | 开发/产品 |

### 3.2 权限列表

| 权限编码 | 名称 | 资源 | 操作 |
|----------|------|------|------|
| project:create | 创建项目 | project | create |
| project:read | 查看项目 | project | read |
| project:update | 编辑项目 | project | update |
| project:delete | 删除项目 | project | delete |
| case:create | 创建用例 | case | create |
| case:update | 编辑用例 | case | update |
| case:delete | 删除用例 | case | delete |
| case:read | 查看用例 | case | read |
| execution:execute | 执行测试 | execution | execute |
| execution:read | 查看执行结果 | execution | read |
| report:read | 查看报告 | report | read |
| report:export | 导出报告 | report | export |
| defect:manage | 管理缺陷 | defect | update |
| user:manage | 管理用户 | user | update |
| role:manage | 管理角色 | role | update |
| system:config | 系统配置 | system | update |
| audit:read | 查看审计日志 | audit | read |

## 四、页面功能详解

### 4.1 用户管理页面

**功能清单：**
- 用户列表（表格：用户名/邮箱/角色/状态/最后登录/操作）
- 搜索（用户名/邮箱模糊搜索）
- 筛选（按角色、状态筛选）
- 创建用户（手动输入 + 邀请链接）
- 编辑用户（修改角色/状态/部门）
- 禁用/启用用户
- 重置密码（发送重置链接邮件）
- 批量导入（CSV模板）

### 4.2 角色管理页面

**功能清单：**
- 角色列表（表格：角色名/编码/描述/用户数/操作）
- 创建角色
- 编辑角色权限（勾选权限树）
- 删除角色（需确认+有用户关联时禁止删除）
- 权限视图：按资源分组展示（功能列表/用例/执行/报告/系统）

### 4.3 团队管理页面

**功能清单：**
- 团队列表（我创建的 + 我加入的）
- 创建团队（名称/描述/初始成员）
- 团队详情（基本信息/成员列表/项目列表）
- 邀请成员（邮箱邀请/邀请链接）
- 成员角色管理（owner/admin/member/guest）
- 移除成员

### 4.4 项目管理-成员权限

**项目内角色：**
- owner：项目创建者，拥有全部权限
- admin：可管理成员、配置项目
- editor：可编辑用例、触发执行
- viewer：只读
