## 一、版本控制

### 1.1 工具

- **版本控制系统**：Git
- **远程仓库**：GitHub (https://github.com/zftest/TestPilot.git)
- **本地路径**：D:/TestPilot/

### 1.2 分支策略

```
main           ← 生产分支，只接受 PR/合并
  └── develop  ← 开发主分支，日常开发在此
        ├── feature/p0-xxx   ← Phase 0 功能分支
        ├── feature/p1-xxx   ← Phase 1 功能分支
        ├── fix/xxx          ← Bug修复分支
        └── docs/xxx         ← 文档更新分支
```

### 1.3 分支命名规范

| 类型 | 格式 | 示例 |
|------|------|------|
| 功能分支 | `feature/phase-{模块名}` | `feature/p0-auth` |
| 修复分支 | `fix/{简短描述}` | `fix/login-redirect` |
| 文档分支 | `docs/{简短描述}` | `docs/api-spec-update` |

### 1.4 Commit 规范

```
{type}: {简短描述}

type 取值：
  feat     = 新功能
  fix      = Bug修复
  docs     = 文档变更
  refactor = 重构
  test     = 测试相关
  chore    = 构建/工具/依赖

示例：
  feat: 实现JWT登录认证接口
  fix: 修复登录页密码框无法粘贴的问题
  docs: 更新PRD v4至v5，补充报表模块细节
```

### 1.5 Tag 规范

```
v{主版本}.{次版本}.{修订号}

示例：
  v0.1.0  ← Phase 0 完成
  v0.2.0  ← Phase 1 MVP
  v0.3.0  ← Phase 2 AI
  v0.4.0  ← Phase 3 全栈
  v1.0.0  ← Phase 4 正式发布
```

## 三、依赖管理

### 3.1 Python 依赖

- 文件：`src/backend/requirements.txt`（精确版本）

### 3.2 Node.js 依赖

- 锁定文件：`package-lock.json`

## 五、发布流程

```
1. develop 分支功能开发完成
2. 本地测试通过
3. 提交 PR：develop → main
4. Code Review（狼图腾确认）
5. 合并到 main
6. 打 Tag（如 v0.1.0）
7. 推送 main + tag 到 GitHub
8. （Phase 4+）GitHub Actions 自动构建 Docker 镜像
```
